# 04 AI Native：为什么 Prompt 不是未来？

> Prompt 不会消失，但它终将退居幕后。
>
> AI Native 的目标，不是让每个人学会写 Prompt，而是让 AI 学会理解每个人。

![image](./images/04-01-cover.png)

---

## 前言

过去两年，如果你接触过 AI，一定绕不开一个词——Prompt。

有人整理 Prompt 模板。

有人出售 Prompt 课程。

有人搭建 Prompt 社区。

甚至有人认为，未来最重要的技能，就是 Prompt Engineering（提示词工程）。

一时间，互联网上充满了各种「万能 Prompt」。

有人希望靠一个 Prompt 写文章，有人希望靠一个 Prompt 做 PPT，还有人希望靠一个 Prompt 搞定所有工作。

如果把时间拨回到 2023 年，这一切都很正常。

因为那时候，大模型刚刚进入大众视野。

Prompt，是人与 AI 沟通最直接、也是几乎唯一的方式。

但仅仅两年过去，我们开始发现一个有趣的现象。

越来越优秀的 AI 产品，反而越来越不强调 Prompt 了。

使用 Cursor 写代码的人，很少再研究复杂的 Prompt。

使用 Perplexity 搜索的人，也不会先写一段长长的角色设定。

越来越多的人，只会说一句：

> “帮我完成这个任务。”

剩下的事情，交给 AI。

于是，一个新的问题出现了。

Prompt 还是未来吗？

我的答案是：

> Prompt 很重要，但 Prompt 不是未来。

因为真正改变软件世界的，从来都不是 Prompt。

而是人与计算机沟通方式的不断演进。

---

## Prompt 为什么会流行？

先说明一点。

这篇文章并不是要否定 Prompt。

恰恰相反。

Prompt 是 AI 发展的必经阶段。

![image](./images/04-02-prompt-engineering-boom.png)

为什么？

因为最早的大模型，并不知道你真正想做什么。

如果你只是输入：

> 帮我写一篇文章。

得到的结果可能非常普通。

于是大家开始不断补充信息。

例如：

* 你是一位资深产品经理。
* 请使用 Markdown 格式。
* 风格要专业、简洁。
* 字数控制在 3000 字左右。
* 最后给出总结。

Prompt 越来越长。

回答越来越好。

于是大家发现：

Prompt 可以提升 AI 的输出质量。

Prompt Engineering 也因此诞生。

从那个阶段来看，它是一项非常有价值的技能。

因为它解决了一个现实问题：

> 如何让 AI 更准确地理解人的意图。

但是，如果我们把时间尺度拉长，就会发现一个更大的规律。

---

## 每一次软件革命，都不是让人学会计算机

![image](./images/04-03-human-computer-evolution.png)

很多人以为，每一次软件革命，都在创造新的软件。

其实不是。

如果回顾计算机的发展历史，你会发现一个非常有意思的现象。

每一次软件革命，都不是让人学会计算机，而是让计算机越来越理解人。

这是我认为理解 AI Native 最重要的一句话。

我们不妨回顾一下过去几十年的软件发展。

---

## 第一阶段：命令行时代

最早使用计算机的人，需要记住大量命令。

打开目录。

复制文件。

删除文件。

进入磁盘。

每一个操作，都需要输入准确的命令。

不会命令，就不会使用电脑。

那个时代，真正适应计算机的人很少。

因为学习成本太高了。

人与计算机沟通，需要先学习计算机的语言。

---

## 第二阶段：图形界面时代

后来，Windows 和 macOS 出现了。

人们第一次发现：

原来不用记命令，也可以操作电脑。

点击图标。

拖动文件。

双击打开。

所有复杂的命令，都被鼠标隐藏了起来。

命令行消失了吗？

没有。

程序员直到今天仍然每天都在使用命令行。

真正变化的是：

普通用户，再也不需要学习命令行了。

计算机开始理解人的点击。

人与计算机沟通的门槛，被大幅降低。

---

## 第三阶段：移动互联网时代

智能手机再次改变了这一切。

以前，我们点击菜单。

后来，我们开始滑动。

缩放。

长按。

手势逐渐替代了复杂的界面操作。

越来越多的人，从来没有学过电脑，却能熟练使用手机。

因为软件开始主动适应人。

而不是要求人适应软件。

---

## 第四阶段：AI 时代

今天，大模型带来了新的交互方式。

我们第一次可以直接使用自然语言。

不用找菜单。

不用点击按钮。

只需要告诉 AI：

> 帮我写一份周报。
>
> 帮我总结这篇论文。
>
> 帮我规划一次旅行。

计算机开始理解人的语言。

人与计算机之间，又少了一层障碍。

看到这里，你会发现一个规律。

软件的发展方向，从来没有改变。

软件发展的历史，本质上就是人与计算机沟通成本不断降低的历史。

命令行如此。

图形界面如此。

触摸屏如此。

AI 也是如此。

---

## Prompt，其实就是 AI 时代的命令行

![image](./images/04-04-prompt-command-line.png)

理解了前面的历史，我们再来看 Prompt。

很多人讨论 Prompt 时，总是在争论：

Prompt 重不重要？

Prompt 会不会过时？

其实，这些问题都没有抓住重点。

Prompt 的真正定位应该是：

> Prompt，是 AI 时代的命令行。

为什么这样说？

因为它们承担的是同一种职责。

命令行时代。

你需要学习计算机的语言。

AI 时代。

你需要学习 Prompt 的表达方式。

本质上，它们都是一种人与计算机沟通的中间语言。

它们当然重要。

但它们从来都不是最终形态。

真正的软件演进方向，一直都是：

> 隐藏复杂性。

Windows 隐藏了命令。

智能手机隐藏了菜单。

那么 AI Native 将会隐藏什么？

答案，就是 Prompt。

---

## 为什么未来最会写 Prompt 的，不是人，而是 AI

理解了 Prompt 在软件发展历史中的位置，我们再来看一个很多人还没有意识到的变化。

如果 Prompt 是 AI 时代的命令行，那么未来，普通用户真的还需要学习它吗？

我的判断是：

> **未来最会写 Prompt 的，不是人，而是 AI。**

这听起来有些反直觉，但事实上，它已经开始发生了。

很多人认为，未来人人都应该学会 Prompt。

但优秀的软件产品，往往会朝着相反的方向演进。

不是让更多人学习底层能力。

而是把底层能力封装起来。

过去如此，现在也是如此。

Windows 没有要求所有人学习 DOS。

智能手机没有要求所有人理解文件系统。

AI Native 也不会要求所有人掌握 Prompt Engineering。

真正优秀的软件，一定会把复杂性留给自己，把简单留给用户。

而 Prompt，就是下一层被隐藏的复杂性。

---

## Prompt 没有消失，只是从前台走向了后台

![image](./images/04-05-prompt-to-background.png)

很多人认为：

> Prompt 会不会消失？

其实，这个问题问错了。

Prompt 不会消失。

真正消失的，是**用户自己编写 Prompt**。

举几个例子。

### Cursor

如果你使用过 Cursor，就会发现一个现象。

很多时候，你只是说一句：

> 修复这个 Bug。

或者：

> 重构这个函数。

剩下的事情，Cursor 会自动完成。

很多人以为：

Cursor 很聪明。

其实，它背后真正发生的是另一件事情。

Cursor 会根据：

- 当前文件
- 项目结构
- 上下文代码
- 用户历史操作
- 当前任务目标

自动生成大量 Prompt。

然后，再交给大模型。

也就是说。

真正写 Prompt 的，不再是用户。

而是 Cursor。

---

### Perplexity

Perplexity 也是一样。

用户只需要输入：

> 帮我分析一下 AI Native 的发展趋势。

但 Perplexity 内部并不是把这句话直接发给模型。

它会：

先搜索。

再筛选网页。

再阅读内容。

再整理引用。

最后重新组织 Prompt。

整个过程，Prompt 比以前更多。

只是全部发生在后台。

---

### NotebookLM

NotebookLM 更明显。

你上传几十份 PDF。

然后问一句：

> 总结一下这些资料。

真正完成工作的，并不是一句 Prompt。

而是 NotebookLM 自动完成：

理解文档。

寻找关联。

组织上下文。

重新生成 Prompt。

最后才得到答案。

所以我们看到一个非常有意思的现象。

> **越优秀的 AI 产品，后台的 Prompt 越复杂；前台的 Prompt 却越来越简单。**

这才是真正的软件演进方向。

---

## AI Native，不是让用户学习 Prompt

![image](./images/04-06-goal-to-execution.png)

如果说过去的软件是在设计功能。

那么 AI Native 软件，更像是在设计目标。

用户真正关心的是：

我要完成什么？

而不是：

我要怎样告诉 AI。

举个例子。

传统软件。

用户要：

打开软件。

找到菜单。

点击按钮。

设置参数。

最后完成任务。

Prompt 时代。

用户变成：

思考 Prompt。

组织 Prompt。

修改 Prompt。

优化 Prompt。

最后完成任务。

AI Native 则完全不同。

用户只需要表达目标。

例如：

> 帮我整理今天的会议内容。
>
> 帮我完成这份周报。
>
> 帮我做一个产品方案。

剩下的事情，由 AI 自己完成。

包括：

理解需求。

生成 Prompt。

调用工具。

执行任务。

检查结果。

整个过程中，用户甚至不知道 Prompt 长什么样。

这就是 AI Native 与 AI 工具最大的区别。

它不是增加了一种新的交互方式。

而是把 Prompt 本身变成了一种内部能力。

---

## 真正改变世界的，从来不是 Prompt

很多人今天讨论 AI，都在讨论 Prompt。

但如果我们把时间拉长，你会发现。

真正改变软件世界的，从来都不是 Prompt。

而是：

**人与计算机沟通方式的改变。**

DOS 时代。

人学习计算机。

Windows 时代。

计算机理解点击。

移动互联网时代。

计算机理解触摸。

AI 时代。

计算机理解语言。

AI Native 时代。

计算机开始理解目标。

注意这里的变化。

过去，我们告诉计算机：

> 我应该怎么做。

未来，我们只告诉计算机：

> 我想完成什么。

至于怎么完成。

由 AI 自己决定。

这是软件发展几十年来最大的变化。

也是 AI Native 真正值得关注的地方。

---

## 结语

如果今天还有人问我：

Prompt 有没有价值？

我的回答依然是：

当然有。

它依然是人与 AI 沟通的重要语言。

它依然会长期存在。

就像命令行今天依然存在一样。

但是，它不会再成为普通用户每天接触的东西。

因为优秀的软件，总是在不断隐藏复杂性。

Prompt 会留下。

但它会越来越像今天的命令行——重要，却隐藏在幕后。

真正站到台前的，将不再是 Prompt。

而是用户的目标。

所以，我更愿意用一句话总结这篇文章：

> **每一次软件革命，都不是让人学会计算机，而是让计算机越来越理解人。**

AI Native 也一样。

它真正改变的，不是 Prompt。

而是人与软件之间的关系。

未来的软件，不会要求你学会写 Prompt。

未来的软件，会努力学会理解你。
