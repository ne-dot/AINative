# 02 视频脚本：AI Native 为什么不是一次功能升级，而是一种新的软件范式？

适用平台：B站、抖音、YouTube  
建议时长：**4-5 分钟**（横屏主版）；竖屏可剪 60-90 秒  
核心结论：**AI Native 不是多一个 AI 功能，而是软件开始替你组织工作。**

本集只讲三点：

```text
1. 范式变了：从功能入口，到目标入口
2. 产品变了：四个能力进入主流程，Prompt 退到后台
3. 入口变了：人和 AI 的交互入口被重新设计
```

## 一、开场：AI 按钮越多，你为什么越累？

时长：0:00-0:25  
画面：`Ep02AiFeaturesAdded` 5s（快速堆满 AI 功能）→ `Ep02WorkflowStillOnUser` 10s（用户流程链还在）→ 字卡

旁白：

> 一条视频讲清楚：为什么 AI Native 不是普通 AI 功能升级，而是更进一步的软件形态。
>
> 如果你也觉得现在到处都是 AI 按钮，但用起来还是累，看这条就够了。
>
> 过去一年，几乎每个软件都多了 AI：写作、摘要、修图、客服。
>
> 但奇怪的是：按钮越来越聪明，你却还是要找入口、选工具、排步骤、检查结果。
>
> 这说明很多 AI 功能只是把某一步变快了，没有把工作交接出去。
>
> 真正的 AI Native，要看一个更狠的标准：**完成工作的责任，到底还在不在你身上。**
>
> 接下来你只要记住三句话：第一，入口从功能变成目标；第二，能力从前台按钮变成后台执行；第三，人和 AI 的交互入口，也会被重新设计。

---

## 二、第一句话：入口从功能变成目标

时长：0:25-0:55  
画面：字卡「第一：从功能到目标」→ `Ep02FeatureUpgrade` 4s → `Ep02GoalCentricOrbit` 8s → `Ep02FunctionVsGoalPath` 15s

旁白：

> 功能升级，是旧软件里多一个能力。比如修图软件多了 AI 填充。
>
> 范式变化，是你不再负责把流程拆开。
>
> 比如你想去掉照片里的路人。传统路径是：选区域、选工具、调参数、修边缘、看结果。
>
> AI Native 的路径应该是：你说“把这个人去掉”，软件自己识别对象、补背景、检查结果。
>
> 一句话：**传统软件的基本单位是功能，AI Native 的基本单位是目标。**
>
> 这句话听起来有点抽象，放到一个真实产品里就很明显了。

## 三、真实产品：Cursor 把目标放到了入口

时长：0:55-2:15  
画面：VS Code + Copilot 侧边栏 3s → Cursor Composer / Agent 录屏

### 录屏任务

Composer 只输入一句：

> 产品列表页的筛选刷新后应该还在，现在一刷新就丢了。修到刷新后筛选仍保持，并在本地验证。

### 分镜

| 画面 | 时长 | 重点 |
|------|------|------|
| Copilot 侧边栏 | 3s | AI 在旁边回答，但主路径仍是你找文件 |
| Cursor 输入目标句 | 5s | 不说改哪个文件，不说用什么方案 |
| Agent 列 todo | 5s | 系统开始拆任务 |
| 读文件 / diff / 终端 | 15s | 快进，表现它在组织工作 |
| 浏览器选择筛选 → 刷新 | 10s | 用户验收结果 |
| 若仍失败，停在失败处 | 5s | 说明验收标准是“任务完成没” |

旁白：

> Cursor 值得拆，不是因为它会写代码，而是因为它把入口改了。
>
> 在很多 IDE 里，AI 像侧边栏助手。它能回答你，但你还是要知道去哪里改、先改什么、怎么验证。
>
> Cursor 的主路径更接近目标：你说“筛选刷新后不能丢”，它去找相关文件、列步骤、改代码、跑命令。
>
> 你没有说“改 URL 参数”，也没有说“改哪个组件”。你给的是结果。
>
> 这就是责任转移：过去是用户拼流程；现在是系统组织流程，用户负责提出目标和验收结果。
>
> 如果它做错了，你也不会问“我是不是少点了一个功能”，你会问：“这个任务完成了吗？”
>
> 但 Cursor 不只是把输入框换成自然语言。它背后真正变化的是：一组能力进入了主流程。

## 四、第二句话：能力从前台按钮变成后台执行

时长：2:15-3:35  
画面：字卡「第二：能力进入后台」→ 四能力字卡，继续用 Cursor 录屏做背景；每个词出现时，对应闪回上一段录屏；最后切“前台一句话 / 后台流程”字卡

旁白：

> 所以 Cursor 不是因为“有聊天框”才更接近 AI Native，而是因为它把四件事接进了主流程。
>
> 第一，**理解目标**。你说的是“筛选刷新后不能丢”，不是“请修改某个 state”。它要先知道你要的结果。
>
> 第二，**读取上下文**。它要知道项目结构、当前文件、相关组件、路由状态、终端报错，而不是每次从空白聊天框开始。
>
> 第三，**调用工具**。它不能只告诉你“可以用 URL 参数保存筛选”，还要能改文件、跑命令、看结果。
>
> 第四，**推进执行**。如果刷新后还是丢，它要继续查、继续改、继续验证，而不是把建议丢给你就结束。
>
> 对应到产品体验上，差别就很明显：侧边栏 AI 更像“给建议的功能”；Cursor Agent 更像“围绕目标组织工作”。
>
> 所以，Prompt 这种交互形式也终将过去。
>
> 不是说 Prompt 消失，而是普通用户不会每天研究复杂 Prompt。
>
> 用户前台只说一句：修复这个 Bug。
>
> 系统后台完成理解目标、读取上下文、调用工具、推进执行。
>
> 越好的产品，越是前台简单、后台复杂。
>
> 当这些能力都藏到后台以后，变化就不只停在某一个软件里。下一个被改变的，是入口本身。

---

## 五、第三句话：人和 AI 的入口会被重新设计

时长：3:35-4:20  
画面：字卡「第三：入口被重新设计」→ OpenAI × Jony Ive 新闻截图 / 官方视频片段 → 手机 App 图标墙 → AI 入口字卡 → 开发者四方向字卡

旁白：

> 当 AI 能理解目标、拿上下文、调工具、跑流程，变化就不只发生在软件内部。
>
> OpenAI 和 Jony Ive 做 AI 硬件，最值得看的不一定是“它是不是下一代手机”。
>
> 真正值得看的是：它是不是想重新设计人和 AI 的入口。
>
> 手机时代的入口是屏幕、App 图标、通知和手指点击。你有需求，第一反应是找 App：聊天打开微信，导航打开地图，写东西打开文档。
>
> AI 硬件如果成立，想改变的可能不是某一个 App，而是这套交互前提：它能不能更自然地感知环境、记住上下文、理解你正在做什么，然后在合适的时候帮你推进任务。
>
> 所以这里不只是“AI 调度几个工具”。更大的变化是：用户第一入口可能从“打开屏幕里的 App”，变成“在当前场景里表达目标”。
>
> App 不一定消失，但它可能从前台入口，变成 AI 背后可调用的能力。

## 六、结尾

时长：4:20-4:35  
画面：结论字卡

旁白：

> 所以，AI Native 不是“软件加了 AI”。
>
> 它真正改变的是分工：用户表达目标，软件组织工作。
>
> 谁能把理解、上下文、工具和执行藏到后台，谁才更接近下一代软件。
